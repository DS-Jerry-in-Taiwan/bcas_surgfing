## 📝 技術決策記錄

### 1. 四層架構設計
**決策**: Scheduler (Go) → Pipeline (Python Step 1-3) → Validators (嵌入 Step 2) → Storage (PostgreSQL)

**理由**:
- 調度器用 Go (非阻塞、性能優)
- 管道用 Python (快速開發、易維護)
- 驗證作為第 2 步的把關 (防止髒數據寫入)
- PostgreSQL 作為唯一存儲源

### 2. Collect-Only 模式
**決策**: 爬蟲不直接寫入 DB，結果存儲在內存，驗證後決定是否寫入

**理由**:
- 防止驗證失敗的數據污染 DB
- 失敗的數據可保存到 logs/validation/failed/ 便於檢查
- 驗證通過才 flush_items() 執行批量寫入
- 可用 --force-validation 跳過驗證限制 (僅開發/測試環境)

### 3. 中心化統計信息
**決策**: 統計信息集中在 src/__version__.py，而非分散在各文檔

**理由**:
- 避免 README 重複
- 支持程序化查詢 (`python src/__version__.py`)
- 單一源 (VERSION + __version__.py)
- 易於版本控制和自動化

### 4. README 聚焦核心內容
**決策**: 移除統計、鏈接、貢獻、許可證章節，專注快速開始和故障排查

**理由**:
- README 是快速參考，應保持簡潔 (537 行)
- 詳細內容分別在其他文件 (CONTRIBUTING.md, LICENSE, etc.)
- 避免信息過載
- 易於維護，減少跨文件依賴

### 5. 非阻塞調度器設計
**決策**: HTTP /run 端點使用 channel + goroutine，立即返回，後台執行

**理由**:
- 避免長連接阻塞
- HTTP 返回 ~1ms
- 管道後台執行 15-20 分鐘
- Webhook 不需要等待結果
