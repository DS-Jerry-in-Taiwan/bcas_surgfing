# CBAS 盤後自動化分析系統 - 高階開發規畫說明書

## 壹、 專案概述與核心目標

本專案旨在建構一套**輕量級、批次處理**的盤後資料管線 (Data Pipeline)。系統將於每日台股收盤後自動啟動，負責蒐集現股與可轉債 (CB) 的報價及籌碼數據，並透過量化模型計算溢價率與隔日沖風險，最終產出一份高可讀性的次日交易戰略清單。

* **核心價值：** 消除人為情緒干擾，精準過濾高風險標的，提升 CBAS 佈局勝率。
* **部署型態：** 終端機應用程式 (CLI) 或排程腳本 (Cronjob)。

## 貳、 系統資料流與模組目標對應

系統運作完全遵照時間軸進行批次處理，以下為每日的標準作業流程 (SOP) 與對應模組：

### 階段一：資料採集 (Data Ingestion)

**⏰ 觸發時間：每日 17:00 (盤後籌碼結算完畢後)**

* **對應模組：盤後批次資料爬蟲模組 (EOD Data Scraper)**
* **模組目標：** 擔任系統的「清道夫」。確保所有分散在各處的原始數據被完整、乾淨地抓取並標準化。
* **執行動作：**
1. 抓取目標清單（如家登、聚和）的現股與 CB 收盤價。
2. 爬取目標現股的「買賣超前五大券商分點」明細與單日總成交量。
3. 獲取 CB 的最新轉換價格（防範除權息異動）與 AS 承作餘額。



### 階段二：價值與型態清洗 (Data Processing)

**⏰ 觸發時間：每日 17:15 (資料採集完畢後)**

* **對應模組：靜態價值與型態運算模組 (EOD Analytics Engine)**
* **模組目標：** 擔任系統的「計算機」。將生硬的報價轉化為 CBAS 實戰指標。
* **執行動作：**
1. 運算核心公式：計算當日收盤後的「轉換價值」與「最終溢價率」。
2. 技術面標記：判斷現股是否滿足「帶量突破」或「站上均線」等攻擊型態。
3. 剔除防護：若溢價率大於 5%，直接標記為廢棄標的，不送入下一關。



### 階段三：籌碼透視與風險定價 (Risk Assessment)

**⏰ 觸發時間：每日 17:20**

* **對應模組：籌碼照妖鏡與風險評級模組 (Chip Profiler & Risk Matrix)**
* **模組目標：** 擔任系統的「防毒軟體」。專門抓出隱藏在紅K棒背後的隔日沖地雷。
* **執行動作：**
1. 黑名單比對：將前五大買超分點與內建的「知名短線客資料庫（如凱基台北）」進行交叉比對。
2. 壓力測試：計算 `潛在短線賣壓張數 ÷ 今日總成交量`，得出「隔日沖風險佔比」。
3. 綜合評分：結合階段二的溢價率與本階段的籌碼健康度，給予該標的 S、A、B、C 等級評分。



### 階段四：戰略輸出 (Reporting)

**⏰ 觸發時間：每日 17:30**

* **對應模組：終端報表輸出模組 (Terminal Report Generator)**
* **模組目標：** 擔任系統的「指揮官」。將複雜的運算結果濃縮成明日開盤的行動準則。
* **執行動作：**
1. 生成視覺化報表（Markdown 或終端機 ASCII 表格）。
2. 列出各標的明日關注重點（例如：綠燈進場區間、紅燈避開提示）。
3. 推播至指定終端（Terminal 畫面、Slack 或 Telegram Bot）。

## 參、 開發流程簡介 (Development Roadmap)

為了確保專案順利落地，建議採用敏捷式的迭代開發，從基礎建設一路推進到自動化部署：

### Phase 0: 環境建置與基礎設施 (Infrastructure Setup)

* **目標：** 建立穩定且可隔離的開發環境。
* **行動：**
1. 初始化 Python 虛擬環境 (使用 `Poetry` 或 `venv`)。
2. 建立 `Dockerfile`，確保未來部署至雲端 (如 GCP Cloud Run 或 Compute Engine) 時環境一致。
3. 選擇並測試資料來源：評估使用爬蟲 (BeautifulSoup/Selenium) 還是付費/開源 API (如 FinMind、Shioaji) 獲取台股盤後資料。



### Phase 1: 資料管線建構 (Data Pipeline Development)

* **目標：** 實作「模組一」，確保資料能穩定流入。
* **行動：**
1. 撰寫爬蟲腳本或 API 串接程式，專注於抓取「收盤價」與「分點進出明細」。
2. 設計輕量級的資料儲存方案（初期可用 JSON 或 SQLite，後續可升級至 PostgreSQL 或 BigQuery）。
3. 加入重試機制 (Retry Mechanism) 與錯誤處理，防範證交所網站阻擋或 API 斷線。



### Phase 2: 核心邏輯實作 (Logic & Analytics Implementation)

* **目標：** 實作「模組二」與「模組三」，賦予系統判斷能力。
* **行動：**
1. 建立 `PremiumCalculator` 類別，封裝轉換價值與溢價率的數學公式。
2. 建立 `RiskAssessor` 類別，載入隔日沖券商黑名單，計算風險佔比。
3. 撰寫單元測試 (Unit Tests)，輸入已知的歷史資料（如昨日的家登與聚和），驗證系統輸出的溢價率與風險燈號是否 100% 正確。



### Phase 3: 介面輸出與排程自動化 (Presentation & Automation)

* **目標：** 實作「模組四」並讓系統全自動運行。
* **行動：**
1. 使用 `Rich` 或類似套件，設計精美的終端機報表輸出格式。
2. (選用) 串接通訊軟體 API (如 Telegram 或 Line Notify)，實現手機推播。
3. 設定系統排程：使用 Linux `crontab` 或雲端服務 (如 GCP Cloud Scheduler)，設定每日 17:30 自動觸發整個 Python 腳本。
